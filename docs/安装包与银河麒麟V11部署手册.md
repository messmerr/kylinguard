# KylinGuard 安装包与银河麒麟 V11 部署手册

本文描述比赛提交物中的“安装包”：它不是源码压缩包，也不是 Windows/桌面
安装程序，而是一个可在目标服务器解压后部署的 Linux 应用包。包内包含
KylinGuard 后端 wheel、已构建的前端静态文件、systemd unit、双账户权限策略、
root 窄口 helper、安装/卸载/核验脚本和本手册；不夹带 Python、Node.js 或第三方
安装包。前端静态产物包含经 Vite 打包的浏览器运行代码，后端第三方运行依赖按
锁定版本从目标机配置的 pip 源安装。

## 1. 目标基线

目标平台是银河麒麟高级服务器操作系统 V11 2503、LoongArch64。方案依据：

- [银河麒麟 V11 2503 官方发布说明](https://www.kylinos.cn/issueNote/serverNote/2066705970141257729.html?activeId=2066705970141257729)
  明确支持 LoongArch64，采用内核 6.6、glibc 2.38、GCC 12.3.1，并提供 base /
  updates 仓库；
- [银河麒麟 V11 系统管理员手册](https://www.kylinos.cn/upload/1/kycms/20250825/1959820093577297920.pdf)
  给出的 Python 基线是 3.11.6，软件包和服务分别使用 dnf、systemd 管理；
- V11 2503 官方 LoongArch64 base 仓库实际提供 Python 3.11.6、GCC 12.3.1、
  Rust/Cargo 1.82、OpenSSL 和 libffi 开发包。

安装器不会添加、删除或改写目标机的 repo 文件，只使用管理员已经配置好的
dnf 和 pip 源。

### 为什么安装器需要编译工具链

截至本方案核对时，PyPI 的 pydantic-core、jiter、cryptography 没有
LoongArch64 通用 wheel，目标机需要从源码构建 Rust/C 扩展。安装器因此通过
dnf 安装 `gcc rust cargo python3-devel openssl-devel libffi-devel pkgconf`。

`cryptography` 47 起最低 Rust 版本提升到 1.83，而 V11 2503 官方仓库是
Rust 1.82，因此安装约束将其固定为 46.0.7；这不是随意降级，而是目标发行版
工具链兼容性约束。最新版 `pydantic-core 2.46.4` 和 `jiter 0.16.0` 的源码
最低 Rust 已升到 1.88，因此分别采用仍满足当前 MCP/FastAPI 接口、且 MSRV
为 1.75 的 `pydantic-core 2.33.2` 和 `jiter 0.10.0`。其余传递依赖也固定到
随本项目完整测试的版本，防止比赛现场在线安装时解析到未经验证的新发布。

这些 Rust 扩展通过 PEP 517 隔离环境构建，构建后端同样必须锁定。
`cryptography 46.0.7` 要求 `maturin>=1.9.4`，因此固定使用仍采用 Rust 2021
Edition、MSRV 1.74 的 `maturin 1.9.4`。安装器把约束文件同时传给 pip 的
`--constraint` 和 `--build-constraint`，避免隔离环境自行解析到要求 Rust 2024
Edition 的新版 maturin。

## 2. 构建安装包

构建机需要 Python 3.11+、Node.js 18+ 和 npm。第三方运行依赖不进入安装包：

```bash
python tools/build_install_package.py
```

输出：

```text
dist/
  KylinGuard-0.1.0-KylinV11-LoongArch64.tar.gz
  KylinGuard-0.1.0-KylinV11-LoongArch64.tar.gz.sha256
```

若 CI 已完成前端构建，可使用：

```bash
python tools/build_install_package.py --skip-frontend-build
```

安装包内部结构：

```text
KylinGuard-<version>-KylinV11-LoongArch64/
  install.sh
  uninstall.sh
  VERSION
  PACKAGE-METADATA.json
  SHA256SUMS
  payload/
    wheels/                 KylinGuard 自身的 py3-none-any wheel
    frontend/               已构建的 Vue 静态文件
  deploy/
    kylinguard.service
    kylinguard.env
    sudoers-kylinguard
    kylinguard-privileged
    constraints-kylin-v11.txt
  docs/
    安装包与银河麒麟V11部署手册.md
    verify-install.sh
```

## 3. 安装

把 `.tar.gz` 和同名 `.sha256` 复制到目标机，先校验外层归档：

```bash
sha256sum -c KylinGuard-0.1.0-KylinV11-LoongArch64.tar.gz.sha256
tar -xzf KylinGuard-0.1.0-KylinV11-LoongArch64.tar.gz
cd KylinGuard-0.1.0-KylinV11-LoongArch64
```

先执行只读预检：

```bash
./install.sh --check-only
```

预检检查：

- 内层 `SHA256SUMS`；
- `nkvers`/`os-release` 中的银河麒麟 V11 证据；
- LoongArch64 架构；
- Python 版本、dnf 和 systemd 基础条件。

正式安装：

```bash
sudo ./install.sh
```

安装器会：

1. 通过现有 dnf 源安装 Python、sudo 和 LoongArch64 源码构建依赖；
2. 创建 `kylinguard` 控制面账户、`kylinguard-exec` 受限执行账户和共享工作组；
3. 在独立版本目录创建 venv，通过当前 pip 源在线安装第三方依赖；
4. 导入检查 FastAPI 应用工厂；
5. 安装 root-owned helper、sudoers 和 systemd unit，并分别使用 `visudo` 校验；
6. 检查控制面不能执行任意 root 命令、执行面不能读取控制面数据库；
7. 启用并启动服务，等待 `/api/health` 返回 200。

使用内网 PyPI 时，显式保留管理员设置的镜像变量：

```bash
sudo --preserve-env=PIP_INDEX_URL,PIP_TRUSTED_HOST ./install.sh
```

若系统依赖已经由镜像预装：

```bash
sudo ./install.sh --skip-system-deps
```

此选项仍会检查所有必需命令和 Rust 最低版本，不会绕过运行条件。

## 4. 访问

服务只监听 `127.0.0.1:8000`，因为当前控制台是单机管理员界面，没有远程登录
认证层：

```text
http://127.0.0.1:8000
```

从管理终端远程访问时使用 SSH 端口转发：

```bash
ssh -L 8000:127.0.0.1:8000 administrator@server
```

不要直接把 8000 端口暴露到业务网或互联网。

## 5. 账户与权限边界

| 身份 | 用途 | 明确不能做什么 |
| --- | --- | --- |
| `root` | 仅安装、systemd 和 helper 内的必要动作 | 不运行 Web/Agent 主进程 |
| `kylinguard` | FastAPI 控制面、审计库、模型凭据 | 不能执行任意 root 命令 |
| `kylinguard-exec` | Shell、文件工具、自定义 MCP 和工作目录操作 | 不能读取数据库、审计、API Key |

关键目录：

| 路径 | 所有者/模式 | 含义 |
| --- | --- | --- |
| `/opt/kylinguard/releases/*` | `root:root`，不可组写 | 版本化程序与前端 |
| `/var/lib/kylinguard` | `kylinguard:kylinguard`，0700 | 数据库、审计、模型/MCP 凭据 |
| `/srv/kylinguard-workspace` | `kylinguard-exec:kylinguard-workspace`，2770 | Agent 可操作区 |
| `/etc/kylinguard/kylinguard.env` | `root:kylinguard`，0640 | 非秘密运行边界 |
| `/etc/sudoers.d/kylinguard` | `root:root`，0440 | 精确账户切换和 helper 入口 |

sudoers 只开放两类能力：

1. `kylinguard -> kylinguard-exec`，目标不是 root，用于让 Agent 工具在受限账户下
   执行；
2. `kylinguard -> root helper`，只允许一个固定路径。

helper 不使用 shell，并在 root 边界重新校验：

- 服务动作只允许 `start/stop/restart` 的 `.service` 单元，且拒绝停启
  `kylinguard.service` 自身；
- 文件清理只允许 `/tmp`、`/var/tmp`、`/var/cache`、`/var/log` 下的普通文件；
- 路径逐级使用 `O_NOFOLLOW` 打开，拒绝符号链接；删除前用
  `renameat2(RENAME_NOREPLACE)` 隔离并核对 inode，拒绝竞态替换。

因此“完全访问”表示在 `kylinguard-exec` 权限范围内开放完整 Agent 能力，不等于
root shell。真正需要 root 的少量动作仍只能通过受约束 helper。

## 6. 部署核验与排障

安装完成后执行：

```bash
sudo /opt/kylinguard/current/docs/verify-install.sh
```

查看服务：

```bash
systemctl status kylinguard.service --no-pager -l
journalctl -u kylinguard.service -n 100 --no-pager
```

常见问题：

- `pip` 构建耗时较长：LoongArch64 首次安装会编译 Rust/C 扩展，属于预期；
- `rustc` 版本不足：确认使用 V11 2503 官方 base/updates repo，且版本至少 1.75；
- 构建 `maturin` 时出现 `edition2024`/`cargo metadata` 错误：说明使用了未修复的
  旧安装包，或安装命令没有应用 `--build-constraint`；请换用重新生成的安装包；
- `python3 -m venv` 失败：确认 `python3`、`python3-pip` 安装完整；
- 服务启动但无首页：核对
  `KG_FRONTEND_DIST=/opt/kylinguard/current/frontend` 和 `index.html`；
- 权限能力页提示 sudo 不可用：执行核验脚本，重点检查 `visudo -cf /etc/sudoers`
  与工作目录组权限。

## 7. 升级、回滚和卸载

重复运行新版安装包会创建新的
`/opt/kylinguard/releases/<version>-<timestamp>`，通过原子更新 `current`
软链接切换版本；数据库、凭据和工作目录位于版本目录之外，不随程序替换。
旧版本默认保留，便于现场回滚。

默认卸载保留配置、数据、凭据、工作目录和专用账户：

```bash
sudo ./uninstall.sh
```

确认不再需要任何数据时才彻底清理：

```bash
sudo ./uninstall.sh --purge --yes
```

彻底清理不可恢复。卸载脚本只允许递归删除五个写死的 KylinGuard 目录，拒绝
变量展开成其他路径。

## 8. 验证边界

仓库自动化可以验证安装包结构、脚本语法、helper 参数/路径边界、前端挂载配置、
wheel 导入和应用测试；最终提交前仍应在比赛同版银河麒麟 V11 2503
LoongArch64 实机或虚拟机完成一次联网安装，并保留以下证据：

- `nkvers` 和 `uname -m`；
- 安装器完整输出；
- `verify-install.sh` 全 PASS；
- `systemctl status` 与 `/api/health`；
- 前端首页、权限能力页、只读诊断和一次需确认的服务操作录屏。

在没有完成同版目标机验证前，不应把 x86_64/WSL 的通过结果表述为
“已完成 LoongArch64 原生适配验证”。
