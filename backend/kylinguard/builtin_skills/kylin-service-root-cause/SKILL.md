# 适用范围

用于服务处于 failed、activating、反复重启、启动后立即退出、找不到可执行文件、权限不足、架构不兼容或依赖单元失败等场景。用户未给出明确单元名时，先请求单元名；不要凭进程名猜测并操作相似服务。

# 强制原则

1. 默认只读。仅“检查、分析、为什么失败”不得规划 start/restart/stop 或文件修改。
2. 状态字段是事实，日志是证据，模型推断必须单独标记。不能把日志中相邻的两条事件自动视为因果关系。
3. 优先寻找时间线上最早的明确失败，而不是只解释最后一条“依赖失败”或“进入 failed 状态”的汇总消息。
4. `systemctl status` 返回 inactive 不等于工具失败；`LoadState=not-found`、`ActiveState=failed`、`Result` 和 `ExecMainStatus` 必须分别解释。
5. 未看到配置内容时不得声称配置语法错误；未检查二进制时不得声称架构不兼容；未取得日志时必须降低置信度。
6. 重启是一项变更，不是诊断捷径。即使用户授权重启，重启成功也只证明当前一次启动成功，不证明原始根因已经消失。

# 工作流

## 1. 确认平台与目标

调用 `kylin.system_identity`，只记录系统、架构和是否匹配目标环境。平台不匹配不会阻止通用 systemd 诊断，但报告中不能称为“银河麒麟原生结论”。

确认服务名满足精确单元语义，例如 `nginx.service`。如果用户只说“数据库”，必须先让用户指定具体单元，或通过只读服务列表取得候选后让用户确认；不要同时诊断多个名称相近的生产服务。

## 2. 取得结构化状态和日志

调用 `kylin.service_diagnosis`，重点读取：

- `LoadState`：单元是否存在并成功加载；
- `ActiveState`/`SubState`：当前生命周期状态；
- `Result`、`ExecMainCode`、`ExecMainStatus`：最近一次运行结果；
- `FragmentPath`、`NeedDaemonReload`、`NRestarts`；
- 日志证据是否完整以及工具生成的 findings。

随后调用 `services.service_status`，用于保留人类可读状态、主进程和最近状态转换证据。两处状态不一致时，以时间更新的证据为准，并说明采集存在时间差。

## 3. 沿错误时间线收敛

先分析 `service_diagnosis` 返回的近期日志。只有日志过多、缺少最早错误或需要验证特定关键词时，才调用 `logs.journal_search`，并使用：

- 精确单元名；
- 与现有证据直接相关的关键词；
- 足够覆盖首次失败、但不过度扩张的时间范围。

将结果归入以下一种或多种证据类别：

1. **单元加载**：not-found、bad-setting、单元文件或 drop-in 加载失败；
2. **执行入口**：No such file、Permission denied、Exec format error、解释器不存在；
3. **进程退出**：明确退出码、信号、超时或 OOM；
4. **依赖关系**：依赖单元有独立失败证据，而非只有“dependency failed”汇总；
5. **资源与环境**：端口占用、磁盘满、内存压力、运行用户权限；
6. **应用配置**：应用自身明确报告配置文件、行号和错误原因；
7. **证据不足**：日志缺失、被轮转、权限不足或输出截断。

## 4. 仅在有架构线索时检查二进制

出现 `Exec format error`、`wrong ELF class`、解释器缺失、迁移后二进制无法启动，或日志明确给出执行路径时，调用 `kylin.binary_compatibility`。路径必须来自本轮状态/日志证据或用户明确提供，不能猜路径。

二进制工具的结论边界：

- `architecture_match=false`：可以确认 ELF 与主机架构不一致；
- `missing_interpreter`：可以确认请求的解释器路径不存在；
- `architecture_match=true`：只能排除明显架构不匹配，不能证明 ABI 和共享库完整；
- 非 ELF：可能是脚本或数据文件，不能继续用 ELF 结论。

## 5. 给出根因层级

结论必须分为：

- **已确认根因**：存在直接错误、对象和结果之间的证据链；
- **高可能原因**：多项证据一致，但仍缺一项直接验证；
- **排除项**：有证据明确不成立；
- **未确认项**：缺日志、权限或工具证据。

如果只有服务 failed 状态而没有明确错误，只能写“确认失败，根因未确认”。

# 变更与复核

只有用户明确要求修复，并且目标、修改内容和影响范围清晰时，才能在本 Skill 之外进入正常变更流程。任何 start/restart、配置写入或权限修改仍须经过 KylinGuard 风险、Reviewer、权限和确认门控。

修复后至少复核：

1. `services.service_status` 显示预期 ActiveState；
2. 新时间窗口内不再出现同一明确错误；
3. 如果原问题是重启循环，观察期内 `NRestarts` 不继续增长；
4. 业务可用性没有证据时，不得只凭 active 写“业务恢复”。

# 停止条件

- 单元不存在且没有可靠候选：停止并请求正确名称；
- systemctl 与 journalctl 均不可用：报告平台/权限缺口并停止；
- 日志权限不足或已轮转：停止根因断言，给出补证建议；
- 需要读取密钥、口令或数据库凭据才能继续：不得展示或尝试收集秘密；
- 用户只要求诊断：在形成只读结论后停止，不执行“试一下重启”。

# 输出格式

1. **当前状态**：单元、ActiveState、SubState、Result、退出码；
2. **时间线证据**：按发生顺序列出关键事件；
3. **根因分级**：已确认、高可能、排除和未确认；
4. **平台/架构证据**：仅在相关时展示；
5. **证据缺口**：日志、权限、截断或观察窗口；
6. **建议动作与风险**：区分只读验证、待确认变更和不建议动作；
7. **复核标准**：说明怎样才能证明问题真正恢复。
