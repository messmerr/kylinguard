# 适用范围

用于“服务器网络不通”“某网卡有丢包”“默认路由异常”“端口访问失败”“服务有没有监听”等问题。它诊断本机网络栈和指定目标的连通性，不代替交换机、DNS、上游防火墙或对端主机的独立证据。

# 强制原则

1. 始终按链路 → 地址 → 路由 → 本机套接字 → 端到端探测的顺序排查，不因用户说“端口不通”就直接修改防火墙。
2. 网络接口错误/丢弃计数通常是启动以来的累计值。只有两个时间点持续增长或已有监控基线时，才能写“当前仍在丢包”。单次非零只能写“存在历史错误或丢弃记录”。
3. `ping` 是主动外发探测，只在用户给出目标或诊断确实需要验证该目标时调用；不得自行扫描网段、枚举主机或扩大目标范围。
4. ping 成功不证明 TCP/UDP 端口可用，ping 失败也可能是 ICMP 被策略丢弃。结论必须限定到被验证的层次。
5. 本 Skill 全程只读，不执行 `ip link set`、`ip route add/del`、`nmcli connection modify`、`firewall-cmd` 变更或配置文件写入。
6. 麒麟原生网络工具被探测到不等于已调用。只有 MCP 输出中存在实际原生证据时才能引用；否则明确说明本轮使用 iproute2/ss 降级链。

# 工作流

## 1. 明确问题范围

从用户描述中提取：

- 受影响接口（若已知）；
- 本机、网关、同网段主机、域名或远端服务；
- 失败方向（入站/出站/双向）；
- 协议与端口；
- 开始时间、是否间歇发生。

用户没有给接口时先全局采集，不要猜 `eth0`、`ens*` 等名称。用户没有给探测目标时，不调用 `network.ping_host`。

## 2. 采集平台和本机网络证据

调用 `kylin.system_identity` 记录平台和架构，只用于报告环境，不让平台结果替代网络证据。

调用 `kylin.network_diagnosis`：

- 指定接口时传精确接口名；未指定时留空；
- 检查 operstate、地址与前缀、MTU、接口 flags；
- 检查 RX/TX errors 和 dropped；
- 检查默认路由、目标网段路由及路由出口接口；
- 保留 `evidence` 中每个采集项是否成功；
- 阅读 `limitations`，不要超出采样能力断言。

如果接口不存在、地址证据为空或路由证据缺失，先报告该层证据缺口，不要跳到应用层结论。

## 3. 判断本机服务监听

当问题涉及“端口访问失败”“服务对外不可达”时调用 `network.listening_ports`，核对：

- 是否存在目标端口；
- TCP/UDP 是否与用户预期一致；
- 监听地址是 `127.0.0.1`、具体地址还是 `0.0.0.0/::`；
- 进程信息是否可见，权限不足时明确说明。

没有监听只能确认“本机当前未观察到目标监听”，不能直接断言应用配置错误。只监听回环地址可以解释远端无法连接，但仍需与实际访问路径对应。

## 4. 有界端到端探测

仅对用户明确给出的域名/IP 调用 `network.ping_host`，count 通常取 2–4。解释规则：

- 收到响应：只证明本轮 ICMP 往返可达；
- 无响应：可能是链路、路由、对端离线或 ICMP 策略，不能单独定位；
- 域名探测失败：如果输出显示解析失败，可确认 DNS/名称解析层存在问题；否则不要臆测 DNS；
- 延迟或丢包：样本很小时只描述本轮观测，不外推长期质量。

当前工具没有端口主动连接能力时，应把 TCP/UDP 端到端状态标为“未验证”，不能用 ping 代替。

## 5. 分层归因

按以下层级形成结论：

1. **链路层证据**：接口 DOWN、MTU/错误/丢弃异常；
2. **地址层证据**：缺少预期地址、前缀不匹配；
3. **路由层证据**：缺默认路由、目标路由指向异常接口；
4. **本机传输层证据**：端口未监听、只绑定回环、协议不匹配；
5. **端到端证据**：指定目标 ICMP 结果；
6. **外部未确认项**：DNS、上游 ACL、交换网络、对端服务。

必须指出最早失败的层。低层已经失败时，高层失败通常是后果，不要重复列为多个独立根因。

# 停止条件

- 目标接口或主机不明确且继续探测会扩大范围：停止并请求目标；
- ip 与 ss 均不可用：报告采集能力缺口并停止；
- 需要扫描网段、端口或第三方主机：不执行，要求用户缩小到授权目标；
- 用户只要求诊断：形成分层结论后停止，不修改网络；
- 只有一次累计错误计数：停止“持续丢包”断言，建议在明确时间间隔后复测。

# 输出格式

1. **故障边界**：接口、方向、目标、协议和时间范围；
2. **分层结论**：链路、地址、路由、本机监听、端到端逐层给出状态；
3. **关键证据**：实际接口、地址、路由、计数器、监听和探测结果；
4. **已确认根因/高可能原因**：说明证据链；
5. **未确认项与限制**：累计计数、外部网络、缺失工具；
6. **下一步验证**：只读验证优先，任何配置修改单独走授权流程。
